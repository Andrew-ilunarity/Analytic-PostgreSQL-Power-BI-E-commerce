# 🚀 Кейс: Наскрізна система операційного та комерційного аналізу для E-commerce (End-to-End)

### 📊 Situation (Ситуація)

Наявний великий масив сирих деструктурованих даних електронної комерції (e-commerce датасет на 250 000+ транзакцій). Дані містили інформацію про замовлення, клієнтів, фінанси, часові мітки та логістичні статуси (повернення). Бізнесу бракувало прозорості: операційний менеджмент не мав чітного розуміння пікових навантажень на склад для планування змін логістів, а комерційний директор та маркетологи не могли відстежувати реальні тренди продажів через щоденну хаотичну волатильність («шум» у даних) та сезонну зміну поведінки покупців.

### 🎯 Task (Завдання)

Розробити та впровадити наскрізну (End-to-End) аналітичну систему. Мені потрібно було:

1. Розгорнути локальну базу даних PostgreSQL, імпортувати датасет та оптимізувати типи даних.
2. Створити складну архітектуру розрахунків на рівні SQL (динамічні погляди/Views) з використанням віконних функцій для агрегації операційних метрик (рухомі середні, кумулятивні підсумки, ранжування категорій).
3. Побудувати інтерактивний двосторінковий дашборд у Power BI для поєднання стратегічного управління (виручка, середній чек, категорійний аналіз) та гнучкого операційного планування (погодинна завантаженість складу, аналітика повернень).

### 🛠️ Action (Дії)

Проект було реалізовано через декомпозицію на чіткі технічні етапи:

* **Етап 1: Інжиніринг даних (SQL/PostgreSQL)**
* Створив базу даних і провів парсинг текстових часових міток за допомогою `TO_TIMESTAMP`.
* Написав конвеєр обчислень із послідовними блоками `WITH` (CTE). Використав віконну функцію `DENSE_RANK() OVER (PARTITION BY... ORDER BY...)` для ізоляції та визначення топ-категорії за витратами для кожного унікального клієнта без стискання деталізації рядків.
* Реалізував розрахунок 7-денного рухомого середнього (`Rolling Average`) через межі вікна `ROWS BETWEEN 6 PRECEDING AND CURRENT ROW`, щоб очистити тренд від щоденних коливань.
* Оптимізував розрахунок накопичувального підсумку (`Running Total`) для фінансового моніторингу за допомогою `SUM() OVER (ORDER BY...)`.
* Побудував окрему модель погодинного розщеплення дат (`EXTRACT(HOUR FROM...)`), виділивши часові компоненти для аналізу внутрішньодобової активності.


* **Етап 2: Бізнес-логіка та розробка мір (DAX/Power BI)**
* Підключив SQL-погляди до Power BI Desktop.
* Написав аналітичні міри на DAX за допомогою змінних `VAR/RETURN`. Застосував функцію `DIVIDE` та контекстні фільтри `CALCULATE` для виведення ключових KPI комерції: Загальний дохід (Total Revenue), Обсяг замовлень, Середній чек (AOV), а також логістичну метрику болю — Рівень повернень (Return Rate %).


* **Етап 3: Проектування інтерфейсу (UI/UX)**
* Розробив гнучку систему зрізів (Slicers) за роками, місяцями та конкретними днями.
* Налаштував візуали так, щоб система миттєво масштабувалася: від річних фінансових звітів для директорів до детального погодинного графіка конкретного дня для начальника складу (визначення годин-пік для виведення додаткових пакувальників та логістів на зміну).



### 🏆 Result (Результат)

* Створено повністю функціональний робочий прототип аналітичної системи класу End-to-End всього за декілька днів (це мій 4-й успішний проект за останні 8 днів).
* **Виявлено приховані бізнес-інсайти:** наприклад, аналіз грудня чітко підсвітив аномальний ріст категорії *Books* (до 25.79% від загального доходу), що дає маркетингу готову стратегію формування подарункових бандлів під новорічний сезон.
* **Оптимізовано логістику:** операційний графік завантаженості складу зафіксував два чітких добових піки (обідній об 11:00 та вечірній о 21:00–23:00 6 грудня 2022 року), що дозволяє компанії економити бюджет на змінах персоналу, уникаючи простою.
* Інтерфейс захищено від перевантаження інформаційним шумом, збережено ідеальний баланс між глибиною даних та простотою сприйняття.
